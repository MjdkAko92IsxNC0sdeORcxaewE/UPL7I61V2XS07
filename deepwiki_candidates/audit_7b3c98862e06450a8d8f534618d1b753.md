## Attacker Path

**preconditions:**
- `USDSGasCompensation` starts at `10e18`
- At least one other trove exists (so GasPool holds ≥ 20e18 total), OR the attacker opens multiple troves to pre-fund GasPool
- TimeLock executes `setUSDSGasCompensation(20e18)` (legitimate governance action; attacker waits for it)

**attacker-controlled inputs:**
- Timing of `closeTrove` call (after TimeLock execution)

**call sequence:**
1. `BorrowerOperations.openTrove(...)` when `USDSGasCompensation = 10e18`
   - Trove debt stored = `netDebt + 10e18`
   - GasPool receives `mint(gasPool, 10e18)`
2. `TimeLock.execute(SystemState.setUSDSGasCompensation(20e18))` — governance, not attacker-controlled, but attacker monitors mempool/events
3. `BorrowerOperations.closeTrove(...)` — attacker calls immediately after step 2
   - `USDS_GAS_COMPENSATION = systemState.getUSDSGasCompensation()` → **20e18**
   - Borrower must supply `debt − 20e18 = (netDebt + 10e18) − 20e18 = netDebt − 10e18`
   - `_repayUSDS(gasPoolAddress, 20e18)` burns **20e18** from GasPool (which only received 10e18 for this trove)
   - Borrower keeps **10e18 USDS** they never repaid

**Net extraction:** borrower originally received `_USDSAmount ≈ netDebt − fee` but only repays `netDebt − 10e18` from their wallet. They retain `≈ 10e18 − fee` USDS. The 10e18 shortfall is silently consumed from other troves' GasPool deposits.

## Why Existing Checks Fail

`_requireSufficientUSDSBalance` at line 575 checks `debt.sub(USDS_GAS_COMPENSATION)` using the **new** 20e18 value, so it actually *lowers* the repayment bar — it does not catch the mismatch. [5](#0-4) 

There is no per-trove storage of the gas compensation amount minted at open time. `USDSGasCompensation` is a single global mutable variable with no snapshot mechanism. [6](#0-5) 

`_getCompositeDebt` and `_getNetDebt` both read the live global, so the stored trove debt (`netDebt + OLD_gasComp`) is permanently mismatched against the live `_getNetDebt(debt)` = `debt − NEW_gasComp` after a raise. [7](#0-6)

## Rejection Checks

**expected behavior checked:** No — the invariant "closeTrove burns exactly what was minted to GasPool at open" is broken. The original Liquity v1 design uses a fixed constant `LUSD_GAS_COMPENSATION`; making it mutable without per-trove snapshots is a deviation that introduces this mismatch.

**prior report checked:** No evidence of this specific root cause (mutable `USDSGasCompensation` without per-trove snapshot) in the indexed codebase or NatSpec.

**README/NatSpec checked:** No NatSpec on `closeTrove` or `setUSDSGasCompensation` documents this interaction or acknowledges the mismatch risk.

**unsupported assumption checked:** The TimeLock execution is a supported, deployed privileged path (`onlyTimeLock` modifier, TimeLock address set in `setConfigs`). The attacker requires no privileged access — only the ability to call `closeTrove` after the governance action executes. [8](#0-7)

## Local Proof Required

**test type:** Foundry unit test

**test file to add:** `test/GasCompMismatch.t.sol`

**test setup:**
```solidity
// Deploy full system with USDSGasCompensation = 10e18
// Fund GasPool with a second trove (so GasPool has ≥ 20e18)
// Attacker opens trove: netDebt = 100e18, compositeDebt = 110e18
// GasPool balance after open: ≥ 20e18 (10e18 from attacker + 10e18 from second trove)
// TimeLock calls SystemState.setUSDSGasCompensation(20e18)
// Attacker calls closeTrove
```

**expected assertion:**
```solidity
// Before close: attacker USDS balance = _USDSAmount (≈ 100e18)
// After close:  attacker USDS balance = _USDSAmount - (110e18 - 20e18) = _USDSAmount - 90e18
//               attacker retains ≈ 10e18 USDS they should have repaid
assertEq(usdsToken.balanceOf(attacker), ~10e18);  // kept, not repaid
// GasPool burned 20e18 but only received 10e18 from attacker's open
// Second trove's 10e18 gas deposit was consumed
assertEq(usdsToken.balanceOf(gasPool), 0);  // second trove's deposit consumed
```

**failure condition:** If `usdsToken.burn(gasPool, 20e18)` reverts (GasPool balance < 20e18), the exploit requires pre-seeding GasPool via additional troves first — still achievable, just requires more setup. The accounting deficit and borrower windfall are confirmed if the burn succeeds.
