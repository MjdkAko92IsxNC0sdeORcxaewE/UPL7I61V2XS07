## Attacker Path

**preconditions:**
- A `BorrowerWrappersScript` instance is deployed (standalone — no DSProxy or delegatecall dispatcher exists anywhere in this repo).
- The contract holds a nonzero BNB balance. This is reachable via:
  - `claimStakingGainsAndRecycle` is `external payable`; BNB sent as `msg.value` or received from `sableStaking.unstake(0)` lands at `address(this)`.
  - `claimSPRewardsAndRecycle` calls `stabilityPool.withdrawFromSP(0, ...)` which sends BNB gains to `address(this)`; if `claimedCollateral == 0` the BNB is never forwarded and stays in the contract.
  - `selfdestruct` force-send from any contract.

**attacker-controlled inputs:**
- `_recipient` = attacker EOA
- `_amount` = `address(this).balance` (full drain)

**call sequence:**
1. Attacker (any EOA) calls `BorrowerWrappersScript.transferBNB(attacker, address(this).balance)`.
2. `_recipient.call{value: _amount}("")` executes with no prior auth check.
3. All BNB transfers to attacker; contract balance becomes zero. [2](#0-1) [3](#0-2) 

## Why Existing Checks Fail

There are **no checks at all**. The function is `external` with no `onlyOwner`, no `msg.sender` validation, no reentrancy guard, and no modifier of any kind.

The original Liquity design used a DSProxy pattern where scripts are called via `delegatecall` — meaning `address(this)` is the proxy's address and the script contract itself never holds funds. However, **this repository contains no DSProxy or delegatecall dispatcher** anywhere in `src/Proxy/` or elsewhere. The script contracts are standalone deployable contracts with `external` functions, making the script contract itself the fund-holding entity. [4](#0-3) 

## Rejection Checks

**expected behavior checked:** In Liquity's original proxy pattern, `transferBNB` is only reachable via delegatecall from a user-owned DSProxy, so the caller context is the proxy owner. That protection is entirely absent here — no proxy dispatcher exists in this repo.

**prior report checked:** The missing-access-control pattern on `transferBNB` is a known class of issue in Liquity forks, but the specific precondition (standalone deployment without DSProxy) is repo-specific and must be verified locally.

**README/NatSpec checked:** No NatSpec or comments on `transferBNB` indicate any expected caller restriction.

**unsupported assumption checked:** The BNB accumulation precondition does not require oracle failure, admin action, or malicious tokens — only a direct call to `claimSPRewardsAndRecycle` or `claimStakingGainsAndRecycle` on the script, or a selfdestruct force-send.
