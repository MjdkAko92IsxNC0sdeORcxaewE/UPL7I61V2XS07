      ‚Üí sableRewarder.issueSABLE()          // issues latestRewardPerSec * elapsed again (if no guard)

Each iteration: `issueSABLE()` re-issues the same `latestRewardPerSec * timeSinceLastIssue` (because `lastIssuanceTime` was not updated), `F_SABLE` increases again, the attacker's snapshot was just reset to the previous `F_SABLE`, so the new delta is non-zero, and the attacker receives another `SABLEGain`.

## Why Existing Checks Fail

**`_requireUserHasStake`** ‚Äî only checks `currentStake > 0`; does not prevent zero-amount unstake calls. [1](#0-0) 

**`_updateUserSnapshots`** ‚Äî resets `F_SABLE_Snapshot = F_SABLE` after each call, which correctly prevents double-claiming the *same* `F_SABLE` delta. However, if `issueSABLE()` increases `F_SABLE` again on the *next* call (because `lastIssuanceTime` was not updated), a fresh non-zero delta is created for the attacker to claim. The snapshot mechanism is not a guard against repeated issuance ‚Äî it only tracks what has already been claimed. [2](#0-1) 

**`CommunityIssuance.issueSABLE` same-block guard** ‚Äî `CommunityIssuance` sets `lastIssuanceTime = block.timestamp` on every call, making `timeSinceLastIssue = 0` on same-block repeats and thus `issuance = 0`. If `SableRewarder` omits this update, the guard is absent. [3](#0-2) 

**`ISableRewarder.issueSABLE()` returns void** ‚Äî unlike `ICommunityIssuance.issueSABLE()` which returns `uint`, the rewarder's interface returns nothing. This is a structural difference suggesting a distinct implementation that may not share the same guard logic. [4](#0-3) 

**Critical blocker:** The concrete `SableRewarder` source is not present in this repository. The glob search for `**/SableRewarder*` returned no `.sol` files, and `source-artifacts/` contains no `sable_rewarder_bscscan.html`. The entire finding is contingent on whether the deployed contract at `0x23d253F1Ab38a1Ec8c05103232B4eFaFB6A1bdEb` omits the `lastIssuanceTime = block.timestamp` update.
