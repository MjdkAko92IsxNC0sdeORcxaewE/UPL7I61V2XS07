## Attacker Path

**preconditions:**
- `USDSGasCompensation = 10e18` at trove open time
- GasPool holds ≥ 20e18 USDS total (i.e., at least one other trove was also opened at 10e18, which is realistic in any live system)
- TimeLock executes `SystemState.setUSDSGasCompensation(20e18)` between open and close (legitimate governance action, not malicious)

**attacker-controlled inputs:**
- Borrower calls `openTrove(...)` when rate = 10e18
- Borrower calls `closeTrove(...)` after TimeLock raises rate to 20e18

**call sequence:**
1. `BorrowerOperations.openTrove(...)` with `USDSGasCompensation = 10e18`
   - Mints 10e18 USDS to GasPool
   - Stores `troveDebt = netDebt + 10e18`
2. `TimeLock.execute(SystemState.setUSDSGasCompensation(20e18))`
3. `BorrowerOperations.closeTrove(...)`
   - Reads `USDS_GAS_COMPENSATION = 20e18` (current)
   - Requires borrower balance ≥ `debt - 20e18` = `(netDebt + 10e18) - 20e18` = `netDebt - 10e18`
   - Burns `netDebt - 10e18` from borrower (10e18 **less** than the net debt they received)
   - Burns 20e18 from GasPool (10e18 **more** than was deposited for this trove)

**Accounting result:**
- Total USDS burned = `(netDebt - 10e18) + 20e18` = `netDebt + 10e18` = original composite debt → USDS supply is arithmetically correct
- But distribution is wrong: borrower retains 10e18 USDS they should have burned; GasPool is over-drained by 10e18 (consuming another trove's gas compensation deposit)

## Why Existing Checks Fail

There is **no per-trove snapshot** of the gas compensation amount at open time. The protocol stores only `compositeDebt` (which embeds the open-time gas comp value) but `closeTrove` recomputes the gas comp split using the **live global** `systemState.getUSDSGasCompensation()`. [7](#0-6) 

The `_requireSufficientUSDSBalance` check at line 575 only verifies the borrower has `debt - current_gasComp`, which is now 10e18 less than the net debt they originally drew — so the check passes and actually enforces the reduced repayment. [8](#0-7) 

There is no guard that compares the stored composite debt against the current gas compensation to detect the mismatch. `_getNetDebt` is a live read with no historical anchor. [6](#0-5) 

## Rejection Checks

**expected behavior checked:** The protocol clearly intends that `closeTrove` burns exactly the gas compensation that was minted at open. The absence of a per-trove snapshot is a design gap, not documented expected behavior.

**prior report checked:** No evidence in the repository of a prior report on this specific open/close gas-comp rate-change mismatch.

**README/NatSpec checked:** No NatSpec or comment acknowledges that a rate change between open and close is safe or intentional.

**unsupported assumption checked:** The TimeLock raising `USDSGasCompensation` is a legitimate, in-scope governance action (`setUSDSGasCompensation` is explicitly callable by TimeLock). The exploit is user-triggered after the governance action — this is not a malicious-admin assumption.

## Local Proof Required

**test type:** Foundry unit test

**test file to add:** `test/GasCompRateMismatch.t.sol`

**test setup:**
```solidity
// Deploy full system with USDSGasCompensation = 10e18
// Open trove A (victim/other trove) → GasPool gets 10e18
// Open trove B (attacker) → GasPool gets 10e18 (total: 20e18)
// Simulate TimeLock: systemState.setUSDSGasCompensation(20e18)
// Record attacker's USDS balance before close
// Call borrowerOperations.closeTrove() as attacker
```

**expected assertion:**
```solidity
// Attacker burned only (netDebt - 10e18) instead of netDebt
// Attacker retains 10e18 extra USDS
assertEq(
    usdsToken.balanceOf(attacker),
    initialBalance - (netDebt - 10e18)  // kept 10e18 extra
);
// GasPool drained by 20e18 (10e18 over its per-trove deposit)
assertEq(usdsToken.balanceOf(gasPool), 0); // 20e18 - 20e18 = 0 (consumed trove A's deposit too)
```

**failure condition:** If `usdsToken.burn(gasPool, 20e18)` reverts (GasPool has insufficient balance), the exploit requires more troves to be open first — adjust setup to open N troves so GasPool ≥ 20e18 before the rate change. If the burn succeeds and the attacker's balance shows the 10e18 surplus, the finding is confirmed as fund extraction.
