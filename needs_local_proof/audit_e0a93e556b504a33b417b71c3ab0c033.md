## Attacker Path

**preconditions:**
- Protocol is in `Status.pythWorking` (Pyth oracle active, `oracleKey = bytes32("PYTH")`)
- A valid, signed Pyth price update exists whose `publishTime` equals the target block timestamp (achievable by submitting the freshest available Pyth VAA in the same block)
- `deviationPyth` as computed by `PriceFeed._getCurrentPythResponse` is 0 — either because the Pyth confidence interval is exactly 0, or because the ratio `conf * DECIMAL_PRECISION / price` truncates to 0 in integer arithmetic when `conf` is very small relative to `price`

**attacker-controlled inputs:**
- `priceFeedUpdateData` — the attacker selects which signed Pyth VAA to submit; they cannot forge signatures but can choose the most favorable valid update

**call sequence:**
1. Attacker monitors Pyth price feed for a VAA where `conf` is 0 or rounds to 0 after `PriceFeed` scaling, and `publishTime == block.timestamp` (or submits in the same second)
2. Attacker calls `TroveManager.redeemCollateral(largeAmount, ..., priceFeedUpdateData)`
3. `priceFeed.fetchPrice(priceFeedUpdateData)` returns `oracleKey = "PYTH"`, `deviationPyth = 0`, `publishTimePyth = block.timestamp`
4. `oracleRateCalc.getOracleRate("PYTH", 0, block.timestamp)` → `absDelayTime = 0`, returns `oracleRate = 0`
5. `_getRedemptionFee(totalBNBDrawn, 0)` → fee = `(floor + baseRate) * totalBNBDrawn / 1e18` (no oracle premium)
6. Attacker receives `totalBNBDrawn - minFee`, saving up to `MAX_ORACLE_RATE_PERCENTAGE` (0.25%) of drawn BNB vs. the intended fee

## Why Existing Checks Fail

There is no floor or minimum on `oracleRate` in `getOracleRate` for the PYTH branch — the function explicitly allows `oracleRate = 0` when both inputs are zero. [4](#0-3) 

`_calcRedemptionRate` simply adds the three components; no guard enforces a minimum oracle premium when Pyth data is present. [5](#0-4) 

`_requireUserAcceptsFee` only checks that the fee is below `_maxFeePercentage`; it does not enforce a minimum oracle-rate component. [6](#0-5) 

## Rejection Checks

**expected behavior checked:** The `IPriceFeed` comment states "if get price from Chainlink, deviationPyth = publishTimePyth = 0" — this documents the Chainlink fallback case. For the Pyth path, non-zero deviation is implicitly expected, but the code does not enforce it. The zero-oracle-rate outcome is not documented as intended behavior. [7](#0-6) 

**prior report checked:** Not found in indexed artifacts.

**README/NatSpec checked:** No NatSpec on `getOracleRate` documents zero as a valid/expected return for the PYTH branch.

**unsupported assumption checked:** The critical unverified assumption is whether `deviationPyth` can realistically be 0 on the PYTH path. This depends on `PriceFeed._getCurrentPythResponse` — specifically how `conf` from the Pyth VAA is scaled to `deviationPyth`. If `conf * DECIMAL_PRECISION / price` can truncate to 0 (e.g., `conf = 1`, `price > 1e18`), the precondition is reachable without forging data. This must be confirmed in `PriceFeed.sol` before promotion.

## Local Proof Required

**test type:** Foundry unit test

**test file to add:** `test/OracleRateZeroDeviation.t.sol`

**test setup:**
1. Deploy `OracleRateCalculation`
2. Deploy `PriceFeedTestnet` and call `setFetchPriceResult(0, block.timestamp, true)` — sets `deviationPyth = 0`, `publishTimePyth = block.timestamp`, `oracleKey = "PYTH"`
3. Deploy minimal `TroveManager` wired to the testnet price feed and `OracleRateCalculation`
4. Open a trove with sufficient collateral; acquire USDS
5. Record `getRedemptionFeeWithDecay(totalBNBDrawn, 0)` vs `getRedemptionFeeWithDecay(totalBNBDrawn, MAX_ORACLE_RATE_PERCENTAGE)`
6. Call `redeemCollateral` with the zero-deviation price feed data

**expected assertion:**
```solidity
assertEq(oracleRateCalc.getOracleRate(bytes32("PYTH"), 0, block.timestamp), 0);
uint feeAtZeroRate = tm.getRedemptionFeeWithDecay(1e18, 0);
uint feeAtMaxRate  = tm.getRedemptionFeeWithDecay(1e18, MAX_ORACLE_RATE_PERCENTAGE);
assertLt(feeAtZeroRate, feeAtMaxRate); // attacker pays less
// BNB received by attacker > BNB received at normal oracle rate
```

**failure condition:** If `PriceFeed._getCurrentPythResponse` always produces `deviationPyth > 0` for any valid Pyth VAA (i.e., Pyth's `conf` is always large enough that the scaled value never truncates to 0), the precondition is unreachable in production and the finding should be REJECT. The local test must use the real `PriceFeed` with a mock Pyth contract returning `conf = 0` or a very small `conf` to confirm the rounding path.
