## Attacker Path

**preconditions:**
- `BorrowerWrappersScript` holds a nonzero BNB balance. This is realistic: `claimCollateralAndOpenTrove` and `claimStakingGainsAndRecycle` are both `external payable` and route BNB through the contract; any dust or leftover from `msg.value`, SP reward claims, or staking gain claims remains in the contract. [3](#0-2) [4](#0-3) 

**attacker-controlled inputs:**
- `_recipient` = attacker-deployed `MaliciousReceiver` contract address
- `_amount` = any value ≤ contract balance

**call sequence:**
1. Attacker deploys `MaliciousReceiver` whose `receive()` calls `BorrowerWrappersScript.transferBNB(address(this), address(target).balance)`.
2. Attacker calls `BorrowerWrappersScript.transferBNB(MaliciousReceiver, X)`.
3. The `_recipient.call{value: _amount}("")` triggers `MaliciousReceiver.receive()`.
4. `receive()` reenters `transferBNB` with the remaining balance.
5. Loop continues until the contract is drained.

## Why Existing Checks Fail

There are **no existing checks**:

- `transferBNB` is `external` with zero access control — no `onlyOwner`, no `msg.sender` restriction, no role check. [5](#0-4) 
- There is no `ReentrancyGuard` or mutex anywhere in `BNBTransferScript` or `BorrowerWrappersScript`.
- There is no DSProxy or `delegatecall` pattern in this codebase (the `src/Proxy/` directory contains standalone deployed contracts, not library-only scripts). The `BorrowerWrappersScript` constructor initializes immutable state, confirming it is deployed and called directly. [6](#0-5) 

**Important nuance:** The reentrancy is technically valid but **redundant**. Because `transferBNB` has no access control at all, an attacker can drain the entire balance in a single non-reentrant call by passing `_amount = address(target).balance`. The reentrancy does not add impact beyond what a direct call already achieves. The root cause is the missing access control; reentrancy is a secondary exploitation path. A local proof should test both vectors.

## Rejection Checks

- **expected behavior checked:** No — `transferBNB` with no caller restriction is not expected behavior for a contract that can hold protocol BNB.
- **prior report checked:** Not found in indexed scope.
- **README/NatSpec checked:** No NatSpec or access-control documentation found on `transferBNB`.
- **unsupported assumption checked:** The precondition (contract holds BNB) is supported by the `payable` functions and protocol flow; no oracle or admin assumption required.

## Local Proof Required

**test type:** Foundry integration test

**test file to add:** `test/BNBTransferReentrancy.t.sol`

**test setup:**
1. Deploy `BorrowerWrappersScript` with mock protocol addresses.
2. Force-send BNB to `BorrowerWrappersScript` (use `selfdestruct` helper or `vm.deal`).
3. Deploy `MaliciousReceiver` whose `receive()` calls `transferBNB(address(this), address(target).balance)`.

**expected assertion:**
```
assertEq(address(borrowerWrappers).balance, 0);
assertEq(address(malicious).balance, initialBalance);
```

**failure condition:** If `BorrowerWrappersScript` retains any BNB after the call, the reentrancy drain did not fully execute. Also test the simpler non-reentrant path: `transferBNB(attacker, balance)` in a single call — if that also drains the contract, the root cause is confirmed as missing access control, and the reentrancy framing is secondary.
