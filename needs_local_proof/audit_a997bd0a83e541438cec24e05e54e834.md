## Exact Code Path

**file:** `src/TroveHelper.sol`
**function:** `requireTCRoverMCR`
**symbols/lines:** lines 129–136

The guard reads:

```solidity
require(
    _getTCR(_price) >= systemState.getMCR(),
    "TroveManager: Cannot redeem when TCR < MCR"
);
``` [1](#0-0) 

In the Liquity reference implementation this check is `TCR >= CCR`. Here it is `TCR >= MCR`. Recovery Mode is defined as `TCR < CCR`, so the window `MCR <= TCR < CCR` passes this guard while the system is already in Recovery Mode. [2](#0-1) 

## Attacker Path

**preconditions:**
- System TCR is in the range `[MCR, CCR)` — i.e., Recovery Mode is active.
- No separate CCR-based guard exists inside `redeemCollateral` in `TroveManager.sol` (unconfirmed; see below).
- Attacker holds USDS and is past the bootstrap period.

**attacker-controlled inputs:**
- `_USDSamount` (redemption size), `_firstRedemptionHint`, `_upperPartialRedemptionHint`, `_lowerPartialRedemptionHint`, `_partialRedemptionHintNICR`, `_maxIterations`, `_maxFeePercentage`

**call sequence:**
1. Price drops such that `MCR <= TCR < CCR` (Recovery Mode).
2. Attacker calls `TroveManager.redeemCollateral(...)`.
3. `TroveManager` calls `troveHelper.requireTCRoverMCR(_price)`.
4. Check passes because `TCR >= MCR`.
5. Redemption proceeds: attacker burns USDS at face value and receives BNB collateral.
6. Total system collateral decreases, TCR falls further below CCR, worsening Recovery Mode.

## Why Existing Checks Fail

`requireTCRoverMCR` is the only TCR-level gate visible in `TroveHelper.sol` for the redemption path. It compares against `getMCR()` rather than `getCCR()`. [2](#0-1) 

`checkPotentialRecoveryMode` exists in `TroveHelper` and correctly uses `getCCR()`, but it is a query function used during liquidation accounting — it is not called as a gate inside the redemption flow. [3](#0-2) 

**Critical uncertainty:** `TroveManager.sol` contains ~26 references to CCR/checkRecoveryMode/Recovery Mode (confirmed by grep). It is possible that `redeemCollateral` itself contains an additional `_checkRecoveryMode` guard that would block the path independently of `requireTCRoverMCR`. This could not be confirmed because the full body of `redeemCollateral` in `TroveManager.sol` was not readable in this pass. If no such guard exists, the finding is valid; if one does, this is a non-issue.
