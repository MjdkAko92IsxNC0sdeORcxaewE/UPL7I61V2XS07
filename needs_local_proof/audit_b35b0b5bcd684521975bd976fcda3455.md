## Attacker Path

**preconditions:**
- Two troves exist: a dust-collateral trove (victim) and a survivor trove (attacker).
- Dust trove has `_coll` (after gas compensation) such that `_coll * DECIMAL_PRECISION + lastBNBError_Redistribution < totalStakes`, i.e. `BNBRewardPerUnitStaked` floors to zero. Concretely: if `totalStakes = 2e18` (2 BNB survivor) and `_coll = 1 wei`, then `BNBNumerator = 1e18 < 2e18`, so `BNBRewardPerUnitStaked = 0`. The dust trove's debt is non-zero (minimum debt), so `USDSDebtRewardPerUnitStaked > 0`.

**attacker-controlled inputs:**
- Liquidation of the dust-collateral trove (any caller can trigger `liquidate`).
- Subsequent `withdrawColl` call on the survivor trove.

**call sequence:**
1. Dust trove is liquidated → `_redistributeDebtAndColl(_debt > 0, _coll = 1 wei)` executes.
2. `BNBRewardPerUnitStaked = 0` → `L_BNB` unchanged; `USDSDebtRewardPerUnitStaked > 0` → `L_USDSDebt` increases.
3. Survivor calls `withdrawColl(X, ...)` → `_adjustTrove` → `applyPendingRewards` → `hasPendingRewards` returns `false` (because `rewardSnapshots[survivor].BNB == L_BNB`).
4. `getTroveDebt(survivor)` returns stored debt (missing the pending USDS debt reward).
5. `vars.newICR` is computed with understated debt → passes `_requireICRisAboveMCR` with a larger withdrawal than the true debt would allow.
6. Survivor receives excess BNB collateral; their on-chain debt remains permanently understated until a future liquidation increments `L_BNB`.

## Why Existing Checks Fail

`hasPendingRewards` is the sole gate for `_applyPendingRewards`. It checks only the BNB accumulator:

```solidity
return (rewardSnapshots[_borrower].BNB < L_BNB);
``` [6](#0-5) 

There is no symmetric check on `L_USDSDebt`. When `BNBRewardPerUnitStaked` rounds to zero, `L_BNB` is not incremented, so the condition is `false` even though `L_USDSDebt > rewardSnapshots[_borrower].USDSDebt`. `getPendingUSDSDebtReward` would return a positive value in this state, but it is never applied to `Troves[_borrower].debt` because the entire `_applyPendingRewards` body is skipped.

`_adjustTrove` then reads `getTroveDebt` (raw stored value) — not `getEntireDebtAndColl` — after the skipped apply, so the ICR check operates on the understated debt. `_requireICRisAboveMCR` and `_requireNewTCRisAboveCCR` both use `vars.newICR` computed from this understated debt. [7](#0-6) 

## Rejection Checks

**expected behavior checked:** The error-correction mechanism (`lastBNBError_Redistribution`) is designed to prevent cumulative loss of BNB precision across multiple liquidations, but it does not prevent a single-event desync between `L_BNB` and `L_USDSDebt`. The asymmetric `hasPendingRewards` check is not documented as intentional behavior for this edge case.

**prior report checked:** No evidence of a prior report for this specific `L_BNB`-only gate in `hasPendingRewards` in the available repository artifacts.

**README/NatSpec checked:** NatSpec on `hasPendingRewards` states "rewards have occurred since the snapshot was made" — the BNB-only check is inconsistent with this stated intent when only debt is redistributed.

**unsupported assumption checked:** The precondition requires a dust-collateral trove (`_coll * 1e18 < totalStakes`). This is the primary feasibility concern. For a system with 1 BNB total stakes, `_coll` must be 0 wei (impossible). For 2 BNB total stakes, `_coll = 1 wei` suffices. Creating a legitimate 1-wei-collateral trove requires an astronomically high BNB price at open time, which is not a realistic assumption. However, the condition could also be triggered if the StabilityPool absorbs most of a liquidation's collateral, leaving a dust remainder for redistribution — this path needs local verification.

    function hasPendingRewards(address _borrower) public view override returns (bool) {
        /*
         * A Trove has pending rewards if its snapshot is less than the current rewards per-unit-staked sum:
         * this indicates that rewards have occured since the snapshot was made, and the user therefore has
         * pending rewards
         */
        if (Troves[_borrower].status != Status.active) {
            return false;
        }

        return (rewardSnapshots[_borrower].BNB < L_BNB);
    }
