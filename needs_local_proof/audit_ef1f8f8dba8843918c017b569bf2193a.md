**symbols/lines:**

- `closeTrove` line 573: reads `systemState.getUSDSGasCompensation()` again (current value, not the value stored in trove debt) → uses it to split user repayment vs. GasPool burn [2](#0-1)

## Attacker Path

**preconditions:**
- TimeLock executes `SystemState.setUSDSGasCompensation(gasComp_2)` where `gasComp_2 > gasComp_1` (the value at trove open time). This is a governance action, not a user action, but it creates the exploit window for any borrower with an open trove.

**attacker-controlled inputs:**
- Timing: borrower opens trove before the gasComp increase and closes after it.

**call sequence:**
1. Borrower calls `openTrove(...)` while `USDSGasCompensation = gasComp_1`.
   - GasPool receives `gasComp_1` USDS (minted).
   - Trove stored debt = `netDebt + gasComp_1`.
2. TimeLock executes `SystemState.setUSDSGasCompensation(gasComp_2)` where `gasComp_2 > gasComp_1`.
3. Borrower calls `closeTrove(...)`.
   - `debt = netDebt + gasComp_1` (stored).
   - `USDS_GAS_COMPENSATION = gasComp_2` (current, from `systemState`).
   - User repays: `debt - gasComp_2 = netDebt + gasComp_1 - gasComp_2 = netDebt - Δ` where `Δ = gasComp_2 - gasComp_1`.
   - GasPool burns: `gasComp_2` (more than it received for this trove).
   - **Borrower keeps `Δ` USDS they never repaid.**

**Concrete numbers (live deployment):**
- Current `gasComp_1 = 10e18`. If TimeLock raises to `gasComp_2 = 20e18`, each borrower who opened at `10e18` and closes at `20e18` extracts `10e18` USDS (10 USDS) per trove, repaying 10 USDS less than they borrowed.

## Why Existing Checks Fail

`closeTrove` has no guard comparing the gasComp embedded in the trove's stored debt against the current `systemState.getUSDSGasCompensation()`. The trove struct stores only `debt` (a single uint), not the gasComp component separately. [4](#0-3) 

`_requireSufficientUSDSBalance` checks `debt.sub(USDS_GAS_COMPENSATION)` using the **current** gasComp, so when gasComp has increased the check is weaker than intended — it requires the user to hold less USDS than they originally borrowed. [5](#0-4) 

`GasPool` is a passive holder with no accounting of per-trove deposits; it cannot detect or prevent the imbalanced burn. [6](#0-5) 

The live-context balance invariant `GasPool.balance == 1 * currentUSDSGasCompensation` holds only because there is currently exactly one trove opened at the current gasComp value. After a gasComp change with open troves, this invariant breaks immediately. [7](#0-6) 

## Rejection Checks

**expected behavior checked:** The Liquity reference design stores gasComp as a fixed constant (`LUSD_GAS_COMPENSATION`). Sable replaced it with a mutable `systemState` parameter without adding per-trove gasComp snapshots or a migration path for open troves. This is not documented as expected behavior for the mutable case.

**prior report checked:** No prior report found in the repository for this specific mismatch between stored trove debt and current gasComp at close time.

**README/NatSpec checked:** `GasPool.sol` NatSpec still references the original fixed-50-USDS Liquity design and does not mention the mutable parameter or transition handling. [8](#0-7) 

**unsupported assumption checked:** The precondition is a TimeLock governance action (`setUSDSGasCompensation`), not a malicious admin key leak. The TimeLock is a live deployed contract at `0x638675b7C2e056917567571307C6f6A7D69A258A`. Governance parameter changes are expected protocol operations; the vulnerability is that the protocol does not handle the transition correctly for already-open troves.

## Local Proof Required

**test type:** Foundry integration test

**test file to add:** `test/GasPoolDrift.t.sol`

**test setup:**
```
1. Deploy full protocol stack (or fork mainnet).
2. Record gasComp_1 = systemState.getUSDSGasCompensation() (10e18).
3. Alice opens a trove: deposits BNB, borrows minNetDebt USDS.
4. Record Alice's USDS balance (= _USDSAmount), GasPool balance (= gasComp_1), trove debt.
5. Prank as TimeLock: call systemState.setUSDSGasCompensation(gasComp_2) where gasComp_2 = 20e18.
6. Alice calls closeTrove().
7. Assert: Alice's USDS balance after close = Alice's balance before close - (trove_debt - gasComp_2).
```

**expected assertion:**
```solidity
// Alice repaid less than she borrowed:
assertEq(aliceUSDSAfter, aliceUSDSBefore - (troveDebt - gasComp_2));
// Alice kept Δ = gasComp_2 - gasComp_1 extra USDS:
assertEq(aliceUSDSAfter - aliceUSDSBefore_postOpen, gasComp_2 - gasComp_1);
// GasPool burned more than it received for this trove:
assertEq(gasPoolBalanceAfter, gasPoolBalanceBefore - gasComp_2); // received gasComp_1, burned gasComp_2
```

**failure condition:** If the test passes (Alice keeps `Δ` USDS and GasPool is over-drained), the finding is confirmed as fund extraction. If `closeTrove` reverts due to insufficient GasPool balance (underflow), the DoS path is confirmed instead and the fund extraction path requires multiple open troves to buffer the GasPool.
