## Attacker Path

**preconditions:**
- At least 2 active troves exist (so GasPool holds ≥ 2× old gas comp, giving it enough balance to cover the inflated payout)
- `USDSGasCompensation` was `oldValue` (e.g. 10e18) when victim's trove was opened; GasPool received exactly `oldValue` for that trove
- Governance (TimeLock proposer) has scheduled `setUSDSGasCompensation(newValue)` with `newValue > oldValue` (e.g. 15e18); the timelock delay has elapsed
- System is in Recovery Mode and victim's trove has ICR between 100% and MCR

**attacker-controlled inputs:**
- Calls `TimeLock.execute(systemState, 0, setUSDSGasCompensation(15e18), ...)` if `EXECUTOR_ROLE` is open to `address(0)`, OR simply waits for anyone to execute it
- Calls `TroveManager.liquidate(victim)` (or `batchLiquidateTroves`) immediately after the parameter change

**call sequence:**
1. `TimeLock.execute(...)` → `SystemState.setUSDSGasCompensation(15e18)` — parameter now 15e18
2. `TroveManager.batchLiquidateTroves([victim])` → `_liquidateRecoveryMode` → reads `systemState.getUSDSGasCompensation()` = 15e18 → sets `singleLiquidation.USDSGasCompensation = 15e18`
3. `_sendGasCompensation` → `usdsToken.returnFromPool(gasPoolAddress, liquidator, 15e18)` — plain ERC20 transfer succeeds because GasPool holds ≥ 15e18 (funded by multiple troves at 10e18 each)
4. Liquidator receives 15e18; only 10e18 was deposited for the victim's trove; 5e18 is drained from other troves' gas compensation reserves [6](#0-5) [7](#0-6) [8](#0-7) 

## Why Existing Checks Fail

1. **No per-trove gas compensation snapshot.** The trove struct stores `debt` (which includes the old gas comp at open time) but there is no stored `gasCompAtOpen` field. At liquidation, `singleLiquidation.USDSGasCompensation` is read live from `systemState` rather than from any per-trove record. [1](#0-0) 

2. **GasPool is a shared unaccounted pool.** `GasPool` is a bare contract with no per-trove balance tracking; `returnFromPool` is a plain ERC20 `_transfer` that only checks the aggregate token balance of `gasPoolAddress`. If GasPool holds ≥ `newValue` in aggregate (from other troves), the transfer succeeds even though only `oldValue` was deposited for the liquidated trove. [3](#0-2) 

3. **`entireTroveDebt` does not self-correct.** The stored trove debt includes the old gas comp (10e18). `debtToOffset + debtToRedistribute = entireTroveDebt` accounts for only 10e18 of gas comp, while 15e18 is paid out. The 5e18 delta is not burned or redistributed — it is simply transferred from GasPool to the liquidator, reducing GasPool's balance below what the remaining troves are entitled to. [9](#0-8) 

4. **TimeLock `execute` is gated by `EXECUTOR_ROLE`, but `onlyRole` passes for `address(0)`.** If the live deployment granted `EXECUTOR_ROLE` to `address(0)`, any unprivileged caller can execute a ready operation. Even if not, the mismatch exists for any liquidation that occurs after a legitimate governance execution. [5](#0-4) 

## Rejection Checks

**expected behavior checked:** The protocol is designed to allow `USDSGasCompensation` to change via TimeLock. There is no documented mechanism to reconcile old deposits with new parameter values. This is not documented as expected behavior.

**prior report checked:** No evidence of a prior report on this specific mismatch in the indexed codebase.

**README/NatSpec checked:** No NatSpec on `_liquidateRecoveryMode` or `_sendGasCompensation` addresses this invariant.

**unsupported assumption checked:** The scenario requires a governance action (TimeLock parameter increase). This is a legitimate protocol operation, not a malicious admin action. The liquidator's exploit is triggered by normal liquidation after the change. The `execute` role configuration is a deployment-time fact that needs live verification.

## Local Proof Required

**test type:** Foundry unit test

**test file to add:** `test/GasCompMismatch.t.sol`

**test setup:**
```solidity
// 1. Deploy full system with USDSGasCompensation = 10e18
// 2. Open victim trove (ICR ~105%, between 100% and MCR=110%) — GasPool receives 10e18
// 3. Open a second trove (to satisfy TroveOwners.length > 1 and fund GasPool)
// 4. Simulate TimeLock execution: directly call systemState.setUSDSGasCompensation(15e18)
//    (in test, grant TimeLock role to test contract or use a mock SystemState)
// 5. Drop price to put system in Recovery Mode with victim ICR between 100% and MCR
// 6. Record liquidator USDS balance and GasPool USDS balance before liquidation
// 7. Call TroveManager.batchLiquidateTroves([victim])
// 8. Record liquidator USDS balance and GasPool USDS balance after liquidation
```

**expected assertion:**
```solidity
assertEq(liquidatorUSDSAfter - liquidatorUSDSBefore, 15e18);  // receives new value
assertEq(gasPoolBefore - gasPoolAfter, 15e18);                // GasPool drained by new value
// Only 10e18 was deposited for victim; 5e18 came from other troves' reserves
assertEq(gasPoolAfter, gasPoolBefore - 15e18);                // net drain > victim's deposit
```

**failure condition:** If the test shows `liquidatorUSDSAfter - liquidatorUSDSBefore == 10e18` (old value), the mismatch does not produce excess payout and the finding is invalid. If the `returnFromPool` call reverts due to insufficient GasPool balance, the attack requires more troves to be open first.
