## Attacker Path

**preconditions:**
- `BorrowerWrappersScript` holds a non-zero BNB balance (e.g., mid-execution residual from `claimSPRewardsAndRecycle` or `claimStakingGainsAndRecycle`, or any BNB sent to the `payable` functions)
- Attacker is any external EOA or contract — no privilege required

**attacker-controlled inputs:**
- `_recipient`: attacker-controlled contract with a `receive()` that re-calls `transferBNB`
- `_amount`: any value ≤ contract balance

**call sequence:**

> Note: reentrancy is technically possible here, but it is not the primary attack vector. The simpler and equally effective attack is a **direct single call**: `transferBNB(attacker_EOA, address(target).balance)`. No reentrancy needed. The missing access control is the root cause.

1. Attacker calls `BorrowerWrappersScript.transferBNB(attackerContract, X)`
2. `_recipient.call{value: X}("")` triggers `attackerContract.receive()`
3. `receive()` calls `transferBNB(attackerContract, address(target).balance)` again
4. Recursion continues until balance is zero
5. All BNB drained to attacker

Or, more directly: attacker calls `transferBNB(attacker, address(target).balance)` in a single transaction — same outcome, no reentrancy required.

## Why Existing Checks Fail

`transferBNB` has **zero access control** — no `onlyOwner`, no `require(msg.sender == owner)`, no `require(msg.sender == address(this))`, no reentrancy guard, and no check that `_amount` is bounded by any accounting variable. [5](#0-4) 

The Liquity upstream design intent is that these Script contracts are used via DSProxy `delegatecall`, where the script bytecode runs in the proxy's storage/balance context and the script contract itself never holds funds. In this deployment, `BorrowerWrappersScript` is a standalone deployed contract with its own balance, breaking that assumption. There is no DSProxy wrapper visible in this repository.

## Rejection Checks

**expected behavior checked:** In Liquity's original architecture, script contracts are stateless and hold no funds (used via delegatecall). This repo deploys `BorrowerWrappersScript` as a direct contract with `payable` functions and `address(this).balance` accounting — the delegatecall assumption does not hold here.

**prior report checked:** The missing-access-control on `transferBNB` in Liquity-fork scripts is a known class of issue, but the specific deployment pattern here (standalone, not DSProxy-wrapped) must be confirmed locally.

**README/NatSpec checked:** No NatSpec or comments on `transferBNB` indicate intentional public access or expected-behavior justification.

**unsupported assumption checked:** No oracle, admin, or external dependency assumption required. The attacker only needs the contract to hold BNB, which is a normal operational state given the `payable` functions and protocol BNB callbacks.

## Local Proof Required

**test type:** Foundry integration test

**test file to add:** `test/BNBTransferDrain.t.sol`

**test setup:**
1. Deploy `BorrowerWrappersScript` with valid protocol addresses (or mock them)
2. Fund the script contract with BNB (e.g., `vm.deal(address(script), 10 ether)`)
3. Deploy `MaliciousReceiver` whose `receive()` calls `script.transferBNB(address(this), address(script).balance)` (for reentrancy path), or simply use an attacker EOA for the direct-drain path

**expected assertion:**
```solidity
// Direct drain (no reentrancy needed):
uint256 before = attacker.balance;
script.transferBNB(attacker, address(script).balance);
assertEq(address(script).balance, 0);
assertEq(attacker.balance, before + 10 ether);
```

**failure condition:** If `transferBNB` reverts or requires caller to be owner/self, the vulnerability does not exist. If the contract never holds BNB in any reachable execution path, the precondition fails and the finding should be rejected.

---

**Key clarification:** The reentrancy framing in the question is technically valid (no guard, external call before any state update) but adds no additional impact over a direct single-call drain. The root cause is the **complete absence of access control** on `transferBNB`, not reentrancy per se. Both paths lead to the same fund extraction outcome. Local proof should test the direct-drain path first as it is simpler and more reliable.
